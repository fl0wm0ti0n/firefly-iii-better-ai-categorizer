# DEC-0021 ‚Äî Additive `directAssign` schema, no migration

- **Status**: Accepted
- **Date**: 2026-06-28
- **Phase**: architecture (US-0007)
- **Work items**: US-0007

## Context

US-0007 introduces a new `directAssign` boolean flag to keyword mappings. When `true`, the mapping bypasses OpenAI and directly assigns the target category. When `false` or missing, existing AI-hint behavior is preserved.

The current schema (`data/category-mappings.json`) has entries with these fields:
- `id`, `name`, `targetCategory`, `keywords`, `enabled`, `created`, `updated`

All existing entries in production have no `directAssign` field.

Options considered:

### Option A: Additive boolean (chosen)

- Add optional `directAssign?: boolean` field to mapping schema.
- Missing key or `undefined` ‚Üí treat as `false` (AI-hint mode).
- `directAssign: true` ‚Üí direct-assign mode.
- No migration script required ‚Äî `JSON.parse` tolerates unknown/missing keys at startup.
- Reader contract: `getDirectAssignment()` checks `mapping.directAssign` (falsy values treated as `false`).
- Writer contract: `addMapping` / `updateMapping` store `directAssign: Boolean(value ?? false)`.

**Pros**:
- Zero migration cost.
- Fully backward-compatible ‚Äî existing entries continue working as AI hints.
- Minimal code change ‚Äî single field check in `getDirectAssignment`.
- Matches existing pattern (e.g., `enabled` field, also additive boolean).

**Cons**:
- Slightly implicit semantics (missing = false), but this is standard JSON-schema practice.
- Operator must manually enable `directAssign` on existing mappings if desired (intended behavior per AC-6).

### Option B: Explicit migration script

- Add a one-time script that reads all mappings, sets `directAssign: false` on each, and rewrites the file.
- Pros: explicit, no ambiguity.
- Cons: extra step, unnecessary write, no functional benefit over Option A.

### Option C: Schema version bump

- Bump `schema_version` from 1 to 2, add migration logic in `loadMappings()`.
- Pros: formal version tracking.
- Cons: overkill for a single additive boolean; `schema_version: 1` already supports additive extensions per DEC-0019.

## Decision

**Option A** ‚Äî additive optional boolean, no migration script, missing key ‚â° `false`.

### Justification

- DEC-0019 (US-0006) established that `schema_version: 1` supports additive extensions without a version bump.
- DEC-0004 (JSON persistence) already accepts that the schema evolves additively.
- The `directAssign` field is a pure additive boolean ‚Äî same pattern as `enabled` (no migration required when `enabled` was first introduced).
- Migration scripts for additive fields add ceremony without value.

## Risks

1. **Spread-merge field leakage**: current `updateMapping` uses `...updates` from `req.body`, allowing arbitrary field injection. **Mitigated** by introducing a whitelist `#MAPPING_FIELDS` in `addMapping` / `updateMapping` (see architecture ¬ß3.1.2).
2. **Operator confusion**: operator may not realize `directAssign: true` means direct-assign mode. **Mitigated** by UI label "Direct assign (bypass AI)" and tooltip explaining the consequence.
3. **Target category validity**: direct-assign may target a category not in Firefly. **Mitigated** by fall-through rule in `#resolveCategory` (architecture ¬ß3.2).

## Alternatives rejected

- Option B (migration script): unnecessary write, no functional benefit.
- Option C (schema version bump): overkill for additive field, violates DEC-0019.

## Closure criteria

- AC-1: data model supports `directAssign` boolean; absent ‚â° AI-hint mode (**covered by Option A**)
- AC-6: all existing mappings without `directAssign` continue as AI hints (**covered by Option A**)
