# DEC-0017: Test Webhook Integration (Move to Unified Panel, Inherit Scope)

## Status
Accepted

## Context
US-0005 requires consolidating categorization controls. Currently, "Test Webhook" lives in
the Maintenance sidebar group, separate from bulk operations. This fragments the workflow:
operator must switch between Categorizer (bulk) and Maintenance (test) panels.

Additionally, the Test Webhook form has a `test-type` select (withdrawal/deposit) that
duplicates the scope concept introduced in DEC-0016.

## Decision
1. **Move Test Webhook form** from Maintenance group into the unified `panel-categorization`
   (DEC-0015).
2. **Remove `test-type` select** — webhook inherits scope from the panel's segmented control
   (DEC-0016).
3. **Additive `scope` field** in `/api/test-webhook` POST body — backward compatible with
   existing `transaction_type` field.

### UI Implementation
```html
<!-- Within panel-categorization -->
<details style="margin-bottom: 15px;">
    <summary>Test Webhook Configuration</summary>
    <div class="test-form">
        <div>
            <label for="test-description">Description:</label>
            <input type="text" id="test-description" value="Test transaction - Purchase at Amazon">
        </div>
        <div>
            <label for="test-destination">Destination:</label>
            <input type="text" id="test-destination" value="Amazon.com">
        </div>
        <!-- test-type select REMOVED — inherits from scope selector -->
    </div>
</details>
```

### Scope → Webhook Type Mapping
| Scope | `transaction_type` sent to backend |
|-------|-----------------------------------|
| `withdrawals` | `withdrawal` |
| `deposits` | `deposit` |
| `both` | `withdrawal` (default for single test) |

### Backend Contract
```javascript
// POST /api/test-webhook
{
    "description": "...",
    "destination_name": "...",
    "scope": "withdrawals" | "deposits" | "both"  // optional; additive
}
```

**Backward compatibility:** `scope` is optional. `transaction_type` still accepted for
existing callers (AC-6 satisfied).

### Backend Logic
```javascript
#onTestWebhook(req, res) {
    const testDescription = req.body?.description || "Test transaction - Purchase at Amazon";
    const testDestination = req.body?.destination_name || "Amazon.com";
    const testTransactionId = req.body?.transaction_id || "test-" + Date.now();

    // Scope-based type (DEC-0017)
    let testType;
    if (req.body?.scope) {
        const scopeMap = { withdrawals: 'withdrawal', deposits: 'deposit', both: 'withdrawal' };
        testType = scopeMap[req.body.scope] || 'withdrawal';
    } else {
        testType = req.body?.transaction_type || "withdrawal";  // backward compat
    }

    // ... rest unchanged
}
```

### Frontend Logic
```javascript
// btn-test-webhook click handler
const scope = document.querySelector('input[name="categorization-scope"]:checked').value;

const response = await fetch('/api/test-webhook', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
        description: testDescription.value.trim(),
        destination_name: testDestination.value.trim(),
        scope: scope  // additive field (DEC-0017)
    })
});
```

## Consequences

### Positive
- Unified workflow: test webhook from same panel as bulk operations
- Scope consistency: webhook uses same scope as bulk (no duplicate controls)
- AC-2 satisfied: all run actions in one panel
- AC-3 satisfied: single scope control (no `test-type` select)
- AC-6 satisfied: additive `scope` field preserves backward compatibility

### Negative
- Removes `test-type` select (breaking change for bookmarked UI state)
- Mitigation: document in user guide (AC-8); scope replaces type selection
- Adds collapsible `<details>` element (minor UI complexity)
- Mitigation: collapsed by default; minimal visual clutter

### Neutral
- Test Webhook moves from Maintenance to Categorizer (logical grouping)
- Default scope for "both" is `withdrawal` (most common test case)
- `transaction_type` field still accepted for backward compatibility

## Alternatives Considered

### Alternative 1: Keep Test Webhook in Maintenance
**Description:** Maintain separate panel; add cross-link to Categorization panel.

**Rejected because:**
- Does not consolidate workflow
- Operator must still switch panels
- AC-2 not satisfied

### Alternative 2: Keep `test-type` Select Alongside Scope
**Description:** Preserve `test-type` select; add scope as separate control.

**Rejected because:**
- Duplicates scope concept (confusing)
- AC-3 requires single scope control
- Scope inheritance is simpler

### Alternative 3: Remove Test Webhook Entirely
**Description:** Rely on real Firefly webhooks for testing.

**Rejected because:**
- Loses quick test capability
- Operator must create real transaction to test
- Test Webhook is valuable for validation
