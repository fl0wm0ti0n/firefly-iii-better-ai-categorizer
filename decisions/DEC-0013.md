# Decision DEC-0013

Status: Accepted

## Context

US-0004 introduces a pending review queue (`data/pending-category-reviews.json`)
that persists across restarts. The queue is accessed concurrently by:
- Webhook handler (adds reviews after pipeline comparison)
- Bulk job processor (adds reviews per transaction)
- REST API `GET /api/reviews` (reads pending list)
- REST API `POST /api/reviews/:id/accept` (mutates status + applies category)
- REST API `POST /api/reviews/:id/reject` (mutates status)

Single-process Node.js with async I/O — no multi-process concurrency, but
overlapping async operations within the event loop.

## Options

1. **In-memory array + write-through** — load on init; mutate in memory;
   `JSON.stringify` + `writeFile` on every mutation; async serialization via
   Node.js event loop
2. **File locking (proper-lockfile)** — explicit lock/unlock around reads/writes;
   safe for multi-process; adds dependency
3. **Atomic write (temp file + rename)** — crash-safe writes; still race-prone
   on concurrent reads
4. **SQLite database** — full ACID; major architectural change; out of scope

## Recommendation

**Option 1: In-memory array + write-through with async serialization.**

Rationale:
- Single-process deployment (DEC-0002) — no multi-process contention
- Node.js event loop serializes async operations within a process
- Follows existing `FailedTransactionService` pattern (load on init, save on
  mutation, JSON file persistence per DEC-0004)
- Zero new dependencies
- Crash between addReview and saveReviews: review lost until next load —
  acceptable for v1 (operator re-categorizes manually)

Trade-offs:
- No crash consistency guarantee (review may be lost if process dies mid-write)
- Not safe for multi-instance deployment (future: SQLite or file locking)
- Large queue (1000+ items): full JSON parse/stringify on every mutation —
  acceptable for personal finance volume

Concurrency pattern:
- All queue operations are `async` — Node.js event loop ensures serialization
- No explicit locks required within single process
- `GET /api/reviews` reads from in-memory array (no file read on every request)

## Impact

- **Positive:** Simple; zero deps; follows existing patterns; fast reads
- **Negative:** Crash may lose pending review; not multi-instance safe
- **Linked:** R-0021, US-0004 (AC-4, AC-5, AC-6), DEC-0004, architecture § US-0004
