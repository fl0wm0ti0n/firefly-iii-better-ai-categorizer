# Decision DEC-0005

Status: Accepted

## Context

BUG-0001: `FireflyService.getCategories()` fails when Firefly returns HTML instead
of JSON, surfacing `Unexpected token < in JSON at position 0` in the admin UI.
Research (R-0007) audited all `FireflyService` fetch paths and confirmed 15+
methods share missing Accept headers and/or blind `response.json()` parsing.

Open question: fix `getCategories()` only (minimal diff) vs extract a shared
`#fetchJson` helper for all Firefly GET/PUT response parsing.

## Options

1. **`getCategories()`-only fix** — add Accept header + content-type guard inline;
   align error message with existing `FireflyException` pattern.
2. **Shared `#fetchJson` helper** — refactor all FireflyService fetch call sites to
   one helper with Accept + content-type validation.
3. **Hybrid** — shared helper introduced in BUG-0001 but adopted only by
   `getCategories()` initially.

## Recommendation

**Option 1: `getCategories()`-only fix for BUG-0001.**

Meets acceptance criteria (`docs/product/acceptance.md` BUG-0001 rows). Smallest
blast radius on an untested monolith (R6). PO discovery explicitly scoped full
service audit out of BUG-0001. Pattern C (`createTransactions` guard) already
exists in-file — copy that approach rather than introduce new abstraction without
tests.

Follow-up: after US-0001 test harness, open a dedicated story to consolidate
patterns A/B/D into a shared helper (reference R-0007 audit table).

## Impact

- **Positive:** Fixes page-load category dropdown failure with minimal regression
  risk; actionable errors for misconfigured `FIREFLY_URL` / PAT.
- **Negative:** Sibling methods (`getTags`, `getTransaction`, etc.) remain
  vulnerable to the same HTML-parse failure until follow-up hardening.
- **Linked:** R-0007, BUG-0001, architecture delta for `#getCategories` error surfacing.
