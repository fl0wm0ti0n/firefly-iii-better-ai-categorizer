# DEC-0016: Segmented Scope Control (Withdrawals / Deposits / Both)

## Status
Accepted

## Context
US-0005 requires a unified transaction scope control. Currently, scope is controlled via:
1. `auto-skip-deposits` checkbox in General Settings (global config)
2. `test-type` select in Test Webhook form (withdrawal/deposit)

These controls are fragmented across panels and do not provide a clear "process deposits only"
option. AC-3 requires a single scope control that expresses expense/income intent.

## Decision
Implement a segmented control (radio group styled as pills) with three options:
- **Withdrawals** — process withdrawals only
- **Deposits** — process deposits only
- **Both** — process both, BUT honor `auto-skip-deposits` from General Settings

### UI Implementation
```html
<div class="scope-selector">
    <label>Transaction Scope:</label>
    <div class="segmented-control">
        <label class="scope-pill">
            <input type="radio" name="categorization-scope" value="withdrawals" checked>
            Withdrawals
        </label>
        <label class="scope-pill">
            <input type="radio" name="categorization-scope" value="deposits">
            Deposits
        </label>
        <label class="scope-pill">
            <input type="radio" name="categorization-scope" value="both">
            Both
        </label>
    </div>
    <p>Honors "Skip Deposits" from General Settings when scope is "Both".</p>
</div>
```

### Backend Contract (Additive `scope` Field)
```javascript
// POST /api/process-uncategorized
{ "scope": "withdrawals" | "deposits" | "both" }  // optional

// POST /api/process-all
{ "scope": "withdrawals" | "deposits" | "both" }  // optional

// POST /api/test-webhook
{ "description": "...", "destination_name": "...", "scope": "withdrawals" | "deposits" | "both" }
```

**Backward compatibility:** `scope` is optional. Absent → existing behavior (`skipDeposits`
config governs). Existing API callers unaffected (AC-6 satisfied).

### Backend Filter Logic
```javascript
async #processUncategorizedTransactions(scope = null) {
    let transactions = await this.#firefly.getAllUncategorizedTransactions();
    const autoConfig = this.#autoCategorizationService.getConfig();

    // Scope override (DEC-0016)
    if (scope === 'withdrawals') {
        transactions = transactions.filter(tx => tx.attributes.transactions[0].type !== 'deposit');
    } else if (scope === 'deposits') {
        transactions = transactions.filter(tx => tx.attributes.transactions[0].type === 'deposit');
    } else if (autoConfig.skipDeposits) {
        // scope === 'both' or null → honor config
        transactions = transactions.filter(tx => tx.attributes.transactions[0].type !== 'deposit');
    }
    // ... rest unchanged
}
```

### Scope → Filter Mapping
| Scope | `skipDeposits` override | Filter behavior |
|-------|------------------------|-----------------|
| `withdrawals` | Force true | Only withdrawals processed |
| `deposits` | Force false; exclude withdrawals | Only deposits processed |
| `both` | Honor saved config | If skipDeposits=true → withdrawals only; if false → all |

## Consequences

### Positive
- Single, visible scope control — no hunting for checkboxes
- Explicit intent: operator chooses withdrawals, deposits, or both
- AC-3 satisfied: unified scope control
- AC-6 satisfied: additive `scope` field preserves backward compatibility
- Aligns with existing `transaction-type-filter` dropdown in Transaction Management panel

### Negative
- Adds optional field to REST contract (minor complexity)
- Requires backend filter logic changes in `App.js`
- Mitigation: additive field; existing callers unaffected

### Neutral
- Default scope is `withdrawals` (most common use case)
- "Both" defers to `auto-skip-deposits` config (preserves existing behavior)
- Test Webhook inherits scope (DEC-0017)

## Alternatives Considered

### Alternative 1: Dropdown Select
**Description:** Use `<select>` dropdown instead of segmented pills.

**Rejected because:**
- Less visible than segmented control
- Requires click to see options
- Segmented pills provide at-a-glance state

### Alternative 2: Per-Action Toggles
**Description:** Add scope toggle next to each button (Process Uncategorized, Process All).

**Rejected because:**
- Duplicates UI elements
- Inconsistent state across buttons
- Single scope control is simpler (AC-3)

### Alternative 3: UI-Only Checkbox Toggle
**Description:** Keep backend unchanged; UI toggles `skipDeposits` config before calling API.

**Rejected because:**
- Implicit contract (UI mutates global config)
- Race conditions if multiple operators
- Explicit `scope` field is cleaner (AC-6)
