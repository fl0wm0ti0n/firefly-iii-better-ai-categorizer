# Decision DEC-0009

Status: Accepted

## Context

US-0003 requires unit tests for `OpenAiService.classify()` schema construction,
UNKNOWN/refusal mapping, and 429 error propagation — without a live OpenAI API key
in CI (R-0017). Research evaluated three mock strategies:

- **A:** Injectable `client` via constructor/`createForTest`
- **B:** `mock.module('openai')`
- **C:** Live API integration (scope_out)

US-0001 already established the injectable-deps factory pattern on `App` (DEC-0006).
Consistency favors the same seam shape on `OpenAiService`.

## Options

1. **Option A — injectable client** — `constructor(deps = {})` with
   `deps.client ?? new OpenAI({ apiKey, maxRetries: 0 })`; add
   `static createForTest(deps)` returning `new OpenAiService(deps)`.
2. **Option B — module mock** — `mock.module('openai', …)` in test files.
3. **Option C — fetch stub** — intercept HTTP at network layer.

## Recommendation

**Option 1 (Option A)** for US-0003.

Tests can assert captured `chat.completions.create` params including
`response_format.json_schema` enum without experimental ESM module mocking.
Production path unchanged when `deps.client` is omitted. Mirrors DEC-0006 naming
(`createForTest`) for maintainer consistency.

`OpenAiException` remains module-private; unit tests assert `err.code === 429` on
caught errors (R-0017 POC verified).

## Impact

- **Positive:** Zero new npm deps; `node --test tests/` picks up
  `tests/openAiService.test.js`; CI needs no API key.
- **Negative:** Small constructor signature change (~5 LOC); narrow test factory
  on production class (same tradeoff as DEC-0006).
- **Linked:** R-0017, US-0003, DEC-0006, DEC-0010.
