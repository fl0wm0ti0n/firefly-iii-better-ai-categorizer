# Decision DEC-0010

Status: Accepted

## Context

US-0003 upgrades the shared OpenAI client from SDK v3 to v4 (`openai@^4`) while
migrating only `classify()` to Structured Outputs (`json_schema`, `strict: true`).
Sibling methods `matchAccount()` and `extractTransactionsFromText()` remain
prompt+parse flows (scope_out per acceptance).

Research R-0015 confirmed all three methods share one constructor — import/init
change is unavoidable even for out-of-scope methods. Alternatives:

- Full Structured Outputs on all methods (scope creep)
- Split into two SDK versions (impossible — single client)
- Defer SDK upgrade until all methods migrate (rejected — AC-1/AC-2 require v4 schema)

## Options

1. **Partial migration** — v4 client for all methods; `json_schema` on `classify()`
   only; `matchAccount` / `extractTransactionsFromText` get v4 call-path swap
   (`chat.completions.create`, drop `.data`) with legacy parse unchanged.
2. **Full migration** — Structured Outputs on all three methods in US-0003.
3. **Classify-only fork** — separate OpenAI client instances per method.

## Recommendation

**Option 1 (Partial migration)** for US-0003.

Meets AC-1/AC-2 without expanding scope. Sibling methods documented as explicit
follow-up tech debt in architecture and sprint handoff. Pin `openai@^4` (not 5.x/6.x)
to limit blast radius per R-0015 POC on 4.104.0.

**Client config:** `new OpenAI({ apiKey, maxRetries: 0 })` — disables SDK built-in
429 retry so `#retryWithBackoff` in `App.js` remains sole app-level retry policy (AC-5).

## Impact

- **Positive:** Smallest correct diff; atomic deliverable (SDK + schema + tests);
  implements deferred DEC-0003 scope.
- **Negative:** `matchAccount` / `extractTransactionsFromText` retain free-text parse
  fragility on v4 client; future story needed for schema-locking those paths.
- **Gates:** `gpt-3.5-turbo` will 400 on `classify()` — acceptable; default remains
  `gpt-4o-mini` (AC-4).
- **Linked:** R-0015, R-0016, US-0003, DEC-0003, DEC-0009.
