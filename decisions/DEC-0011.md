# Decision DEC-0011

Status: Accepted

## Context

US-0004 requires fetching per-account transaction history from Firefly III to
compute category dominance. Each webhook event or bulk-job transaction triggers
a history fetch. Without caching, a heavy account (2000–5000 transactions) would
require 4–10 paginated API calls per webhook — unacceptable latency and rate-limit
risk per R-0018.

## Options

1. **In-memory cache with 1-hour TTL** — `Map<accountId, {data, timestamp}>`;
   lost on process restart; acceptable staleness for history patterns
2. **File-based cache** — persist to `data/account-history-{id}.json`; survives
   restarts; adds disk I/O and file-locking complexity
3. **No cache** — fetch on every webhook/bulk transaction; always fresh; high
   API load
4. **Background refresh** — cron-style scheduler; decoupled from webhook; adds
   scheduling complexity

## Recommendation

**Option 1: In-memory cache with 1-hour TTL.**

Rationale:
- Single-process deployment model (DEC-0002) — no multi-instance inconsistency
- History patterns are stable; 1-hour staleness acceptable for dominance calc
- Zero dependencies; follows existing in-memory patterns (JobList)
- Bulk mode optimization: fetch once per account at batch start, reuse across
  transactions in same batch
- Restart cost: first webhook after restart pays fetch cost once; acceptable

Trade-offs:
- Cache lost on restart → first webhook pays cold-fetch latency (1–10 API calls)
- Not suitable for multi-instance deployment (future: file-based or Redis)
- Manual invalidation requires process restart (future: API endpoint)

## Impact

- **Positive:** Webhook latency reduced from 4–10 API calls to 0 (cache hit);
  bulk mode fetches once per account
- **Negative:** Cold start after restart; no cross-instance consistency
- **Linked:** R-0018, US-0004 (AC-1), architecture § US-0004
