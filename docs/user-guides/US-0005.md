# US-0005: Auto-Categorization Panel

## Purpose

US-0005 consolidates the admin UI by merging three separate categorization panels (Bulk Categorization, Batch Jobs, Individual Jobs) into a single unified **Auto-Categorization** panel. The Test Webhook form moves from the Maintenance group into this panel. A new segmented scope control (Withdrawals / Deposits / Both) unifies transaction type selection across all run actions. General Settings were also folded into this panel.

## Prerequisites

- **Categorizer running**: The Firefly III AI Categorizer service must be running and connected to your Firefly III instance
- **OpenAI API key**: Configured via `OPENAI_API_KEY` environment variable for AI classification
- **Firefly III access**: Valid API token configured via `FIREFLY_III_ACCESS_TOKEN`
- **Skip Deposits setting**: If you want to exclude deposits when scope is "Both", enable "Skip Deposits" inside the Auto-Categorization panel

## Usage Steps

### 1. Access the Categorization Panel

1. Open the categorizer web interface (default: `http://localhost:3000`)
2. In the sidebar navigation, click **"Auto-Categorization"** under the Categorizer group
3. The unified panel displays:
   - Transaction Scope selector (segmented control)
   - Run actions (Process Uncategorized, Process All)
   - Test Webhook Configuration (collapsible)
   - Job Monitor (batch jobs and individual jobs)

### 2. Select Transaction Scope

The scope control determines which transactions are processed:

- **Withdrawals** (default): Process only withdrawal transactions (expenses)
- **Deposits**: Process only deposit transactions (income, refunds, transfers in)
- **Both**: Process all transactions, honoring the "Skip Deposits" setting from General Settings

To change scope:
1. Click the desired scope pill button
2. The selected scope is highlighted with a filled background
3. All run actions (Process Uncategorized, Process All, Test Webhook) use this scope

### 3. Run Bulk Categorization

#### Process Uncategorized Transactions

1. Select the desired scope (Withdrawals / Deposits / Both)
2. Click **"Process Uncategorized Transactions"**
3. Confirm the action in the dialog
4. The system processes only transactions without categories
5. Live progress appears in the Job Monitor section below
6. Batch controls (Pause, Resume, Cancel) are available while processing

#### Process All Transactions

1. Select the desired scope
2. Click **"Process All Transactions (Overwrite Categories)"**
3. Confirm the warning dialog (this overwrites existing categories)
4. The system processes all transactions within the selected scope
5. Live progress appears in the Job Monitor section below
6. Batch controls are available while processing

### 4. Test Webhook

The Test Webhook form simulates a Firefly III webhook call to test AI categorization without creating real transactions.

1. Expand the **"Test Webhook Configuration"** section (collapsed by default)
2. Enter a transaction description (e.g., "Purchase at Amazon")
3. Enter a destination name (e.g., "Amazon.com")
4. The webhook inherits the scope from the Transaction Scope selector:
   - **Withdrawals** → tests withdrawal categorization
   - **Deposits** → tests deposit categorization
   - **Both** → defaults to withdrawal for single test
5. Click **"🧪 Test Webhook Categorization"**
6. The result appears in the Individual Jobs section of the Job Monitor

### 5. Monitor Job Progress

The Job Monitor displays two sections:

#### Batch Jobs

- Shows all bulk processing jobs (Process Uncategorized, Process All)
- Each job displays:
  - Status (Running, Paused, Cancelled, Completed)
  - Progress bar with percentage
  - Statistics (Processed, Successful, Errors)
  - Batch controls (Pause, Resume, Cancel)
  - Error details (expandable)
- Jobs update in real-time via WebSocket
- Completed jobs remain listed for review

#### Individual Jobs

- Shows single transaction processing (Test Webhook results)
- Each job displays:
  - Transaction description
  - Categorization result (category assigned)
  - Confidence score (if available)
  - Processing time
- Jobs update in real-time via WebSocket

### 6. Batch Controls

While a batch job is running, you can:

- **Pause**: Temporarily stop processing (click Pause)
- **Resume**: Continue processing from where it stopped (click Resume)
- **Cancel**: Stop processing permanently (click Cancel)

Controls appear as buttons within the batch job card.

## Limitations

- **Scope is UI-only**: The scope control affects only the unified panel's run actions. Direct API calls to `/api/process-uncategorized`, `/api/process-all`, and `/api/test-webhook` remain backward compatible (scope field is optional)
- **Test Webhook defaults**: When scope is "Both", the Test Webhook defaults to "withdrawal" for single test (cannot test both types simultaneously)
- **Skip Deposits interaction**: When scope is "Both", the "Skip Deposits" setting inside the Auto-Categorization panel determines whether deposits are processed
- **No per-job scope override**: Each bulk run uses the panel's current scope; you cannot specify different scopes for different jobs in the same session
- **Job history lost on restart**: In-memory job history is lost when the categorizer service restarts (existing limitation)

## Troubleshooting

### Scope selector not responding

- **Symptom**: Clicking scope pills does not change selection
- **Cause**: JavaScript error in browser console
- **Solution**: Refresh the page; check browser console for errors

### Batch job stuck in "Running" state

- **Symptom**: Progress bar not updating, status remains "Running"
- **Cause**: Network issue or service interruption
- **Solution**: 
  1. Check browser console for WebSocket errors
  2. Refresh the page
  3. Check categorizer service logs
  4. Cancel the job and restart

### Test Webhook returns error

- **Symptom**: "Error: [message]" alert after clicking Test Webhook
- **Cause**: Missing fields or backend error
- **Solution**:
  1. Ensure all fields are filled (description, destination)
  2. Check browser console for detailed error
  3. Verify OpenAI API key is configured
  4. Check categorizer service logs

### Process All overwrites categories unexpectedly

- **Symptom**: Existing categories are replaced
- **Cause**: "Process All" overwrites all categories by design
- **Solution**: Use "Process Uncategorized" to only categorize transactions without categories

### Scope "Both" skips deposits unexpectedly

- **Symptom**: Deposits are not processed even though scope is "Both"
- **Cause**: "Skip Deposits" is enabled inside the Auto-Categorization panel
- **Solution**: 
  1. Look for the "Skip Deposits from categorization" checkbox near the top of the Auto-Categorization panel
  2. Uncheck it and click "Save General Settings"
  3. Retry the bulk operation

### Sidebar shows old panel names

- **Symptom**: Sidebar shows "Bulk Categorization", "Batch Jobs", "Individual Jobs" instead of single "Auto-Categorization" entry
- **Cause**: Browser cache
- **Solution**: Hard refresh (Ctrl+F5 or Cmd+Shift+R)

## Technical Notes

### Backward Compatibility

All REST endpoints remain backward compatible:

- `POST /api/process-uncategorized` — accepts optional `scope` field
- `POST /api/process-all` — accepts optional `scope` field
- `POST /api/test-webhook` — accepts optional `scope` field (maps to `transaction_type`)

Existing API callers that do not send `scope` continue to work with existing behavior (governed by `skipDeposits` config).

### Scope Filter Logic

| Scope | skipDeposits Config | Filter Behavior |
|-------|---------------------|-----------------|
| `withdrawals` | Ignored | Only withdrawals processed |
| `deposits` | Ignored | Only deposits processed |
| `both` | Honored | If `skipDeposits=true` → withdrawals only; if `false` → all transactions |
| (absent) | Honored | Existing behavior preserved |

### DOM Structure

The unified panel contains:

```
panel-categorization
├── Scope selector (segmented control: Withdrawals / Deposits / Both)
├── Run actions
│   ├── Process Uncategorized button
│   └── Process All button
├── Test Webhook Configuration (collapsible <details>)
│   ├── Description input
│   ├── Destination input
│   └── Test Webhook button
└── Job Monitor
    ├── Batch Jobs sub-container (id="batch-mount")
    └── Individual Jobs sub-container (id="mount")
```

Socket.io events route to the correct containers automatically (no handler changes required).
