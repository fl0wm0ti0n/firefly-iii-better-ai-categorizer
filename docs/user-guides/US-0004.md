# US-0004: Review Queue Workflow

## Purpose

The review queue workflow enables operators to review and approve AI categorization suggestions before they are applied to transactions. This human-in-the-loop approach ensures accuracy by comparing account history patterns with AI predictions and presenting discrepancies for manual review.

## Prerequisites

- **Categorizer running**: The Firefly III AI Categorizer service must be running and connected to your Firefly III instance
- **Transactions categorized**: At least some transactions in your Firefly III instance should already have categories assigned (for history analysis)
- **Account history available**: The account being categorized must have sufficient transaction history (minimum 10 categorized transactions by default)
- **OpenAI API key**: Configured via `OPENAI_API_KEY` environment variable for AI classification

## Usage Steps

### 1. Access the Review Panel

1. Open the categorizer web interface (default: `http://localhost:3000`)
2. Look for the **"Pending Reviews"** entry in the sidebar navigation
3. The badge next to "Pending Reviews" shows the count of items awaiting review
4. Click the entry to open the review panel

### 2. Review Pending Items

The review panel displays a list of transactions that require manual review. Each item shows:

- **Transaction details**: Date, description, amount, and account
- **History suggestion**: Category suggested by account history analysis with confidence percentage (e.g., "Groceries (90%)")
- **AI suggestion**: Category predicted by OpenAI with confidence percentage (e.g., "Restaurants (75%)")
- **Recommendation**: Which suggestion the system recommends (history or AI) based on higher confidence

### 3. Accept a Suggestion

To apply the recommended category to a transaction:

1. Review the transaction details and both suggestions
2. Click the **"Accept"** button next to the recommended category
3. The system will:
   - Apply the chosen category to the transaction in Firefly III
   - Add the "ai-categorized" tag to the transaction
   - Mark the review as "accepted" in the queue
   - Remove the item from the pending reviews list

### 4. Reject a Suggestion

To dismiss a suggestion without applying any category:

1. Review the transaction details
2. Click the **"Reject"** button
3. The system will:
   - Mark the review as "rejected" in the queue
   - Leave the transaction uncategorized in Firefly III
   - Remove the item from the pending reviews list

### 5. Monitor Queue Health

- **Pending count**: Displayed at the top of the review panel
- **Oldest item age**: Shows how long the oldest pending review has been waiting (helps identify bottlenecks)
- **Queue persistence**: Pending reviews are saved to `data/pending-category-reviews.json` and survive service restarts

## Example

**Scenario**: A transaction at "REWE Markt" for €45.50

**Review panel displays**:
```
Transaction: 2026-06-14 | REWE Markt | -€45.50 | Account: Checking

History Suggestion: Groceries (90%)
  Based on 18/20 previous transactions at this account

AI Suggestion: Restaurants (65%)
  OpenAI prediction based on transaction description

Recommendation: History (Groceries) - Higher confidence
```

**Action**: Click "Accept" on the History suggestion to categorize as "Groceries"

## Limitations

- **Minimum transaction threshold**: Account history analysis requires at least 10 categorized transactions (default). Accounts with fewer transactions will skip history analysis and rely solely on AI classification.

- **Dominance threshold**: History suggestions are only made when one category represents ≥80% of the account's categorized transactions (default). Accounts with diverse spending patterns may not trigger history-based suggestions.

- **Account-level only**: History analysis operates at the account level, not per-merchant. All transactions on an account are analyzed together, regardless of payee.

- **No auto-categorization**: The system never automatically applies categories without operator approval. All suggestions require explicit Accept or Reject action.

- **Single recommendation**: When both history and AI provide suggestions, the system recommends the one with higher confidence. In case of a tie, history is preferred.

- **No bulk operations**: Each review must be accepted or rejected individually. Bulk accept/reject is not currently supported.

- **Queue growth**: If reviews accumulate faster than they are processed, the queue can grow large. Monitor the pending count and oldest item age to prevent bottlenecks.

## Troubleshooting

### Review Queue Not Loading

**Symptoms**: Review panel shows "Loading..." indefinitely or displays an error

**Solutions**:
1. Check that the categorizer service is running: `docker ps | grep categorizer`
2. Verify the data directory is accessible: `ls -la data/pending-category-reviews.json`
3. Check service logs for errors: `docker logs <container-name>`
4. Restart the categorizer service if necessary

### Accept/Reject Failing

**Symptoms**: Clicking Accept or Reject returns an error or has no effect

**Solutions**:
1. Verify Firefly III API connectivity: Check `FIREFLY_III_URL` and `FIREFLY_III_PAT` environment variables
2. Test Firefly III API manually: `curl -H "Authorization: Bearer <PAT>" <FIREFLY_III_URL>/api/v1/accounts`
3. Check that the transaction still exists in Firefly III (it may have been deleted)
4. Review categorizer logs for API errors (401, 403, 404, 500)
5. Verify the PAT has write permissions for transactions

### History Not Available

**Symptoms**: Review panel shows only AI suggestions, no history suggestions

**Causes**:
- Account has fewer than 10 categorized transactions (below `minTransactionCount` threshold)
- No single category represents ≥80% of transactions (below `dominanceThreshold` threshold)
- Account history cache is cold (first request after service restart)

**Solutions**:
- Categorize more transactions manually in Firefly III to build history
- Adjust thresholds in `data/dominance-config.json` (requires service restart):
  ```json
  {
    "dominanceThreshold": 0.70,
    "minTransactionCount": 5
  }
  ```
- Wait for cache to populate (1-hour TTL, automatic refresh)

### Queue Growing Too Large

**Symptoms**: Pending count exceeds 100+ items, oldest item age > 7 days

**Solutions**:
- Process reviews in batches (dedicate time daily to clear the queue)
- Review and adjust dominance threshold if too many items are queued
- Consider whether hard rules (account mapping, auto-categorization) can be configured to reduce review volume
- Archive old reviews by moving them from `data/pending-category-reviews.json` to a backup file

### Configuration Not Taking Effect

**Symptoms**: Changes to `data/dominance-config.json` are ignored

**Solutions**:
- Restart the categorizer service (config is loaded at startup)
- Verify JSON syntax is valid: `jq . data/dominance-config.json`
- Check service logs for config loading messages: `docker logs <container> | grep "History analysis config"`
- Ensure file permissions allow reading: `chmod 644 data/dominance-config.json`
